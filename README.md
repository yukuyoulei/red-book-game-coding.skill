# red-book-game-coding

将《红宝书》方法论转化为游戏开发中的可执行流程，帮助团队在复杂需求下做出更稳健的技术与版本决策。

## 核心目标

- 以事实驱动决策，避免“拍脑袋方案”
- 抓住主要矛盾，减少多线并行导致的内耗
- 通过阶段推进与复盘机制，降低返工率

## 适用场景

- 需求很多，但资源有限，不知道先做什么
- 玩法方向不确定，需要低成本试错
- 跨职能协作（程序/策划/美术/QA）沟通割裂
- 版本推进频繁返工，缺少阶段目标和复盘闭环

## 不适用场景

- 一次性低风险小修复
- 纯 API/工具查询任务
- 紧急线上止血（应先恢复服务，再复盘）

## 固定执行流程（5 步）

1. 调查研究：收集事实，区分现象与根因  
2. 矛盾拆解：列出矛盾点，锁定 1 个主要矛盾  
3. 群众路线：把一线反馈转成可验证假设  
4. 阶段推进：给出本周/本版本/后续储备计划  
5. 批评与自我批评：输出继续做、停止做、下次验证指标  

## 标准输出结构

- Facts（事实面板）
- Contradictions（矛盾优先级）
- Phased Plan（阶段作战图）
- Risk/Alignment（风险与协作对齐）
- Retrospective（复盘条目）

## 快速使用

可以直接在对话中这样触发：

```text
请用 red-book-game-coding 方法，分析我们当前新手引导流失问题，
按 Facts / Contradictions / Phased Plan / Risk / Retrospective 输出，
并给出本周可落地的 1-2 项行动。
```
